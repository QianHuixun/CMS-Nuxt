# 项目实施计划

## 1. 实施目标

基于 [项目需求.md](/home/junxun/my-cms/chunyu-cms-v2/docs/项目需求.md) 的 6 个一级模块，完成后台菜单、页面、数据表、接口与前端展示所需的管理能力建设。

本次实施以“先可上线、后做增强”为原则，优先完成：

- 后台菜单与页面可访问
- 关键业务数据可录入、可编辑、可删除、可展示
- 人才与论文、课题、专利的关联关系可闭环保存
- 文件、图片、视频上传能力优先复用现有组件
- 自动化能力先做基础版，不做复杂算法平台化

## 2. 当前项目现状

结合当前仓库，现状如下：

- 后台管理端已具备 Vue 3 + Element Plus 基础框架。
- 服务端已具备 Nuxt 4 + Drizzle ORM + SQL migration 能力。
- `资源导航管理` 已有菜单与占位页。
- `系统基础配置管理`、`人才与对应科研管理`、`论文著作与成果管理`、`自动化辅助功能`、`活动管理` 已有菜单 migration 和占位页。
- 当前业务接口目录仍较少，大部分新增模块尚未建立正式的 `api/admin` 业务接口。
- 项目已具备可复用的上传与编辑基础能力：
- `ImageUpload` 可用于图片上传
- `FileUpload` 可用于普通文件上传
- `VideoUpload` 可用于视频上传、分片上传、封面帧截取
- `RichTextEditor` 可用于沉浸式编辑的第一阶段实现
- 后台已有头像裁剪实现，可复用于人才照片裁剪

## 3. 实施原则

- 一级标题对应父菜单，二级标题对应子菜单。
- 以 [项目需求.md](/home/junxun/my-cms/chunyu-cms-v2/docs/项目需求.md) 为唯一业务基线。
- 先完成 MVP 闭环，再做增强能力。
- 优先复用现有组件与基础设施，不新增重复技术方案。
- 后台数据结构优先标准化，避免在页面中直接堆叠非结构化字段。
- 关联关系优先用中间表设计，不在主表中直接冗余存储多值字符串。

## 4. 总体实施阶段

### 阶段 0：需求对齐与结构校正

目标是把需求文档和当前项目结构对齐，输出统一实施口径。

交付内容：

- 明确 6 个父菜单和对应子菜单
- 明确每个模块的 MVP 与第二阶段边界
- 确认当前菜单命名与主需求文档的差异
- 确认“人才管理”与“专家库管理/课题时空管理”之间的映射关系

完成标准：

- 输出统一实施计划
- 开发、产品、测试对模块边界理解一致

### 阶段 1：菜单、路由、占位页整理

目标是让需求中的全部模块在后台可进入、可承载后续功能开发。

交付内容：

- 校验现有菜单 migration 是否与主需求一致
- 修正不一致的菜单命名、排序、权限标识
- 补齐遗漏的页面入口
- 保证全部菜单可跳转，不出现 404

完成标准：

- 后台中 6 个父菜单全部可见
- 子菜单与主需求文档一致
- 页面占位能正常访问

### 阶段 2：基础数据模型与通用接口建设

目标是把后续模块共用的数据结构先搭起来。

交付内容：

- 设计并创建各业务主表
- 建立人才与成果的关联表
- 建立统一分页列表、详情、新增、编辑、删除接口规范
- 建立排序、上下架、精选标记等通用操作接口

完成标准：

- 数据结构能支撑 6 个模块的第一阶段功能
- 后端接口具备统一命名与返回格式

### 阶段 3：核心业务模块开发

目标是优先完成“配置类 + 数据录入类 + 关联类”核心模块。

优先顺序建议：

1. 系统基础配置管理
2. 资源导航管理
3. 人才与对应科研管理
4. 论文著作与成果管理

完成标准：

- 关键数据可录入、修改、删除、查看
- 关联关系可保存并可回显
- 文件上传、图片裁剪、视频封面处理可用

### 阶段 4：自动化与内容运营模块开发

目标是补齐自动化辅助和活动内容管理能力。

交付内容：

- 词云基础版
- 年度图表统计
- 沉浸式编辑第一阶段
- 头条精选配置

完成标准：

- 运营类内容可以通过后台完成配置和发布
- 自动化能力可以基于已有数据生成基础结果

### 阶段 5：联调、验收与发布

目标是完成后台、接口、前端展示联调并准备上线。

交付内容：

- 功能联调
- 数据回归验证
- 文件上传链路验证
- 菜单权限验证
- 验收测试

完成标准：

- 核心流程全部通过
- 无阻断性问题
- 可以进入正式部署阶段

## 5. 模块实施计划

### 5.1 系统基础配置管理

对应菜单：

- 父菜单：系统基础配置管理
- 子菜单：导航配置
- 子菜单：屏保配置

MVP 实施内容：

- 导航配置支持维护首页 3 个入口的名称、简介、跳转链接、排序、启用状态
- 屏保配置支持上传开屏视频或图片
- 屏保配置支持设置无人操作触发时长
- 支持预览当前配置

数据设计建议：

- `system_nav_config`
- `screen_saver_config`

页面形态建议：

- 导航配置使用表单加列表
- 屏保配置使用表单加媒体预览

依赖说明：

- 复用 `ImageUpload`
- 复用 `VideoUpload`

### 5.2 资源导航管理

对应菜单：

- 父菜单：资源导航管理
- 子菜单：资源录入
- 子菜单：卡片管理

MVP 实施内容：

- 资源录入支持资源名称、简介、封面图、访问链接、演示视频、显示状态
- 卡片管理支持列表、删除、显示隐藏、排序调整
- 排序第一阶段可先使用排序值或上移下移，不强制要求拖拽

数据设计建议：

- `resource_navigation`

页面形态建议：

- 资源录入用标准 CRUD 表单
- 卡片管理用列表页加排序操作

依赖说明：

- 复用现有资源导航页面目录
- 复用图片和视频上传组件

### 5.3 人才与对应科研管理

对应菜单：

- 父菜单：人才与对应科研管理
- 子菜单：人才管理

MVP 实施内容：

- 人才基础信息 CRUD
- 支持上传高清照片
- 支持照片裁剪
- 支持在人才页面通过下拉选择已存在的论文、课题、专利
- 保存后建立人才与成果的关联关系
- 编辑时能够正确回显已关联成果
- 明确不允许在人才页面直接新增论文、课题、专利

数据设计建议：

- `talent`
- `paper`
- `topic`
- `patent`
- `talent_paper`
- `talent_topic`
- `talent_patent`

页面形态建议：

- 人才列表页
- 人才新增/编辑弹窗或独立表单页
- 论文、课题、专利使用可搜索下拉多选

关键实现说明：

- 人才与成果关系为多对多
- 关联关系通过中间表保存
- 保存人才时同步维护关联表

依赖说明：

- 复用裁剪能力
- 成果下拉数据依赖论文、课题、专利主数据已存在

### 5.4 论文著作与成果管理

对应菜单：

- 父菜单：论文著作与成果管理
- 子菜单：数据导入
- 子菜单：书籍仿真录入

MVP 实施内容：

- 数据导入支持论文、专利、课题三类批量导入
- 导入结果支持分页展示
- 支持论文精选标记
- 支持专利视频上传
- 支持自动生成封面帧或手动上传封面
- 书籍仿真录入支持基础信息、封面、封底、核心章节图片上传

数据设计建议：

- `paper`
- `topic`
- `patent`
- `book_simulation`
- `book_simulation_page`

页面形态建议：

- 数据导入页采用导入入口加结果列表
- 书籍仿真录入采用主表单加图片集管理

关键实现说明：

- Excel 导入优先复用现有 Excel 服务能力
- 视频压缩如第一阶段实现成本高，可先完成上传、封面生成、MP4 校验

依赖说明：

- 复用 `FileUpload`
- 复用 `VideoUpload`
- 复用现有 Excel 能力

### 5.5 自动化辅助功能

对应菜单：

- 父菜单：自动化辅助功能
- 子菜单：词云
- 子菜单：图表生成

MVP 实施内容：

- 词云支持提取单篇论文关键词
- 支持展示所有收录论文关键词
- 支持人工调整关键词权重
- 图表生成功能支持按年度统计论文数量并生成折线图

数据设计建议：

- `paper_keyword`
- 统计结果可先动态查询，不强制落表

页面形态建议：

- 词云页展示关键词列表与权重调整区
- 图表页展示筛选条件和图表预览

关键实现说明：

- 第一阶段词云可基于人工关键词字段或摘要切词结果
- 第一阶段图表以后台查询聚合为主，不先做定时任务

依赖说明：

- 依赖论文数据结构化录入完成
- 后台图表可直接使用现有 `echarts`

### 5.6 活动管理

对应菜单：

- 父菜单：活动管理
- 子菜单：沉浸式编辑
- 子菜单：头条精选

MVP 实施内容：

- 沉浸式编辑支持图文混排
- 支持文字、图片、表格、视频的增删改
- 支持内容随时更新
- 支持视频上传、封面帧截取和手动封面上传
- 头条精选支持活动置顶、排序、上下架状态

数据设计建议：

- `activity`
- `activity_content`
- `activity_headline`

页面形态建议：

- 沉浸式编辑第一阶段优先采用富文本编辑器
- 头条精选采用列表加开关加排序

关键实现说明：

- 第一阶段的“所见即所得”以富文本方式落地
- 更复杂的模块化页面搭建能力放到第二阶段

依赖说明：

- 复用 `RichTextEditor`
- 复用图片和视频上传能力

## 6. 技术实施建议

### 后端

- 在 `server/db/schema` 下新增业务 schema
- 使用 Drizzle migration 管理表结构变更
- 在 `server/services` 中建立对应业务服务层
- 在 `server/api/admin` 下建立各模块接口目录

建议目录：

- `server/api/admin/systemBaseConfig/`
- `server/api/admin/resourceNavigation/`
- `server/api/admin/talentResearch/`
- `server/api/admin/paperAchievement/`
- `server/api/admin/automationTools/`
- `server/api/admin/activityManagement/`

### 后台前端

- 保持现有 `src/views` 目录分组方式
- 列表页统一采用查询区、表格区、操作区结构
- 编辑页优先使用弹窗表单，复杂内容用独立页面
- 文件上传、图片上传、视频上传优先复用现有组件

### 数据与权限

- 每个子菜单至少具备 `list`、`add`、`edit`、`remove` 权限
- 状态切换、排序、精选可拆成独立权限
- 删除优先采用逻辑删除，避免误删造成前台数据缺失

## 7. 风险与重点关注项

- 当前主需求文档与已有菜单命名存在差异，需要先统一口径。
- 人才关联成果依赖论文、课题、专利主数据先录入完成。
- 视频“自动压缩”涉及服务端处理链路，第一阶段建议降级为上传加封面处理。
- 词云如果没有成熟分词方案，第一阶段需允许人工维护关键词。
- 书籍仿真录入只是后台录入能力，前台翻页效果应单独联调。
- 沉浸式编辑如果要求完全还原前端布局，第一阶段富文本方案可能不够，需要提前确认边界。

## 8. 验收标准

- 6 个父菜单和全部子菜单在后台可访问。
- 每个模块 MVP 功能至少完成一条完整录入到展示的闭环。
- 人才与论文、课题、专利的关联可新增、可回显、可修改。
- 论文、课题、专利可导入并参与后续模块使用。
- 词云和图表能够基于真实录入数据产出结果。
- 活动内容可编辑并具备头条精选能力。
- 上传、裁剪、视频封面等链路稳定可用。

## 9. 建议开发顺序

建议按以下顺序推进：

1. 菜单与页面口径统一
2. 系统基础配置管理
3. 资源导航管理
4. 论文著作与成果管理
5. 人才与对应科研管理
6. 自动化辅助功能
7. 活动管理
8. 联调验收与发布

这样安排的原因是：

- 先完成基础配置和资源录入，便于形成最早可见成果
- 先完成成果主数据，再做人才关联，依赖更清晰
- 自动化功能依赖前面数据沉淀
- 活动管理适合在基础数据与上传链路稳定后再推进
