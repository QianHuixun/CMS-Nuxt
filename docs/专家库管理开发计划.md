# 专家库管理开发计划

## 1. 背景与目标

当前后台已存在 `人才与对应科研管理 / 专家库管理` 菜单和 Vue 页面，但页面仍使用 `mockData` 驱动，新增、编辑、删除、导入、导出都没有真实接口和持久化能力。

本计划目标是把专家库管理从静态原型升级为可用的业务模块：

- 支持专家基础信息的新增、编辑、查询、详情、删除、启停和排序。
- 支持专家高清照片上传，后续可接入裁剪能力。
- 支持专家与已存在的论文、课题、专利建立关联关系。
- 编辑专家时能正确回显已关联成果。
- 不在专家库页面内直接新增论文、课题、专利，只选择已有成果主数据。
- 保持当前项目的中后台 UI 风格和 Nuxt + Drizzle 后端实现方式。

## 2. 当前代码现状

### 2.1 菜单与页面

菜单由 `chunyu-cms-web/server/db/migrations/0014_add_planned_admin_menus.sql` 添加：

- 父菜单：`人才与对应科研管理`
- 子菜单：`专家库管理`
- 前端组件：`talentResearch/expertLibrary/index`
- 权限标识：`talentResearch:expertLibrary:list`

当前页面文件：

- `chunyu-cms-admin/src/views/talentResearch/expertLibrary/index.vue`

页面已有的 UI 骨架：

- 搜索区：专家姓名、研究方向、职称、状态。
- 操作区：新增专家、导入、导出、批量删除。
- 表格区：多选、姓名、头像、职称、研究方向、机构、电话、状态、创建时间、操作。
- 弹窗表单：新增/编辑专家。
- 详情弹窗：专家基础字段展示。
- 分页组件：复用 `Pagination`。

当前不足：

- 列表数据来自 `mockData`。
- 查询条件没有真正过滤后端数据。
- 新增、编辑、删除只显示成功提示，不保存。
- 导入、导出只显示“开发中”。
- 缺少专家照片上传/裁剪控件。
- 缺少论文、课题、专利关联选择。
- 缺少后端 API、Drizzle schema、SQL migration。

### 2.2 相关需求约束

已有需求文档对该模块有明确边界：

- 专家/人才页面录入学者个人简历和高清照片。
- 高清照片可作为前端卡片背景。
- 支持照片裁剪。
- 支持选择已录入的论文、课题、专利。
- 保存专家后，数据库自动建立专家与成果的关联关系。
- 不允许在专家页面直接录入新论文、课题、专利。

### 2.3 UI 风格理解

项目后台是典型 Element Plus 中后台形态：

- 左侧深色导航 + 顶部工具栏 + 白色内容区。
- 页面以搜索表单、操作按钮、数据表格、弹窗表单为主要结构。
- 视觉优先级是信息密度、可维护性和低学习成本。
- 当前业务页面普遍使用 `el-card shadow="never"` 包裹搜索区和表格区。
- 操作按钮使用 Element Plus 图标，如 `Plus`、`Search`、`Refresh`、`Delete`、`Upload`、`Download`。
- 状态信息使用 `el-tag`。
- 详情信息使用 `el-descriptions`。
- 大批量信息使用表格展示，长文本使用 `show-overflow-tooltip`。

专家库管理应继续沿用这种低装饰、强操作、表格优先的后台风格，不单独做营销式卡片页。

## 3. 功能范围

### 3.1 MVP 必做

#### 列表与查询

- 分页查询专家列表。
- 支持按姓名模糊搜索。
- 支持按研究方向筛选。
- 支持按职称筛选。
- 支持按状态筛选。
- 支持按创建时间倒序、排序字段升序展示。
- 空数据时展示 Element Plus 表格空态。

#### 新增与编辑

- 新增专家。
- 编辑专家。
- 表单字段校验。
- 编辑时回显专家基础信息、图片和已关联成果。
- 提交后刷新列表。

#### 详情

- 查看专家详情。
- 展示基础信息、简介、联系方式、照片。
- 展示关联论文、课题、专利的摘要信息。

#### 删除与启停

- 单条删除。
- 批量删除。
- 删除前二次确认。
- 支持启用/停用状态切换。
- 删除采用逻辑删除，保留历史关联数据可追溯。

#### 照片

- 支持上传头像/高清照片。
- 第一阶段复用现有 `ImageUpload` 组件。
- 存储图片 URL 或相对路径，不存二进制。
- 预留裁剪能力接入点。

#### 关联成果

- 关联已有论文。
- 关联已有课题。
- 关联已有专利。
- 使用可搜索多选下拉。
- 保存专家时，在同一事务内同步维护关联表。
- 编辑专家时，正确回显已关联 ID。
- 关联成果被删除或停用后，详情页和编辑页要能降级显示。

### 3.2 第二阶段

- Excel 导入专家。
- 导出专家列表。
- 专家照片裁剪。
- 专家卡片预览，展示高清照片作为背景。
- 专家审核状态流转。
- 专家排序拖拽。
- 与前台展示页打通。

### 3.3 暂不做

- 不在专家库页面新增论文。
- 不在专家库页面新增课题。
- 不在专家库页面新增专利。
- 不做复杂审核工作流。
- 不做前台展示页重构。
- 不做复杂组织机构树管理。

## 4. 数据设计

### 4.1 `talent`

专家主表。字段建议基于 `docs/数据库字段规划.md`，结合当前页面字段落地。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `talent_id` | `int` | 主键，自增 |
| `name` | `varchar(100)` | 姓名，必填 |
| `name_en` | `varchar(150)` | 英文名 |
| `gender` | `char(1)` | 性别：`1` 男，`2` 女，`3` 未知 |
| `avatar` | `varchar(500)` | 头像/高清照片 |
| `card_background` | `varchar(500)` | 卡片背景图 |
| `organization` | `varchar(200)` | 所属机构 |
| `department` | `varchar(200)` | 院系/部门 |
| `title` | `varchar(100)` | 职称 |
| `position` | `varchar(100)` | 职务 |
| `research_direction` | `varchar(500)` | 研究方向 |
| `profile` | `text` | 个人简介 |
| `email` | `varchar(150)` | 邮箱 |
| `phone` | `varchar(50)` | 联系电话 |
| `sort` | `int` | 排序 |
| `status` | `char(1)` | 状态：建议沿用业务字段规划，`0` 启用，`1` 停用 |
| `del_flag` | `char(1)` | 删除标记：`0` 未删除，`1` 已删除 |
| 通用字段 |  | `remark/create_by/create_time/update_by/update_time` |

兼容说明：

- 当前前端 mock 使用 `status: '1'` 表示启用、`'0'` 表示禁用。
- 现有字段规划使用 `status: '0'` 表示启用、`'1'` 表示停用。
- 实施前需要统一状态枚举。建议新业务模块采用字段规划约定：`0` 启用，`1` 停用，并同步调整前端标签。

### 5.2 API 约定

#### 分页列表

`GET /api/admin/talent/expert/list`

参数：

- `pageNum`
- `pageSize`
- `name`
- `researchDirection`
- `title`
- `status`

返回：

```json
{
  "rows": [],
  "total": 0
}
```

#### 详情

`GET /api/admin/talent/expert/:id`

返回专家基础信息和关联成果：

```json
{
  "talentId": 1,
  "id": 1,
  "name": "张明华",
  "papers": [],
  "topics": [],
  "patents": []
}
```

#### 新增

`POST /api/admin/talent/expert`

请求体包含基础字段和关联 ID：

```json
{
  "name": "张明华",
  "title": "教授",
  "researchDirection": "中医基础理论",
  "paperIds": [1, 2],
  "topicIds": [3],
  "patentIds": []
}
```

#### 编辑

`PUT /api/admin/talent/expert`

必须包含 `id` 或 `talentId`。

#### 删除

`DELETE /api/admin/talent/expert/:id`

支持逗号分隔批量 ID，或前端循环请求。建议后端兼容 `1,2,3`。

#### 状态切换

`PUT /api/admin/talent/expert/status`

```json
{
  "ids": [1],
  "status": "0"
}
```

### 5.3 服务层重点

`ExpertLibraryServices` 需要包含：

- `pageList(params)`
- `getById(id)`
- `add(payload)`
- `update(payload)`
- `delete(ids)`
- `changeStatus(ids, status)`
- `syncRelations(tx, talentId, relationPayload)`

实现要求：

- 新增和编辑必须使用事务。
- 编辑时先更新专家主表，再删除旧关联，最后插入新关联。
- 删除专家时优先逻辑删除主表，并清理或保留关联表需统一约定。建议逻辑删除主表，关联表保留，查询时过滤主表 `del_flag`。
- 查询列表默认排除 `del_flag = '1'`。
- 返回数据同时补充 `id: talentId`，匹配前端通用表格习惯。

## 10. 风险与依赖

### 依赖

- 论文、课题、专利主数据表和下拉接口。
- 图片上传服务 `/api/admin/common/upload`。
- 后续照片裁剪组件或复用用户头像裁剪能力。

### 风险

- 当前 mock 页面状态枚举与字段规划不一致，需要先统一。
- 成果主数据未完成时，专家关联功能只能预留。
- 导入导出涉及 Excel 字段映射和错误回传，建议放到第二阶段。
- 如果前台要展示专家卡片，后台字段需要提前保留背景图、排序、状态。

## 11. 推荐优先级

1. 统一状态枚举和字段命名。
2. 完成专家主表 CRUD。
3. 接入照片上传。
4. 等论文、课题、专利主数据可用后，接入关联关系。
5. 最后做导入导出和裁剪增强。

这样可以先让“专家库管理”从 mock 页面变成真实可维护数据，再逐步补齐成果关联和运营效率能力。
