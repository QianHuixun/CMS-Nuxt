# 课题时空管理开发计划

## 1. 背景与目标

当前后台已经存在 `人才与对应科研管理 / 课题时空管理` 菜单、静态设计稿和 Vue 页面骨架，但实际页面仍由 `mockData` 驱动，新增、编辑、详情、删除只停留在前端提示层，没有后端接口和数据库持久化。

本计划目标是把课题时空管理从静态原型升级为可用的业务模块：

- 支持课题基础信息的新增、编辑、查询、详情、删除和状态维护。
- 支持按时间录入课题，管理课题级别、负责人、来源、摘要和进度。
- 作为专家库关联成果之一，为专家与课题绑定提供稳定主数据。
- 保持当前项目中后台 UI 风格和 Nuxt + Drizzle 后端实现方式。

## 2. 当前代码现状

### 2.1 菜单与页面

菜单由 `chunyu-cms-web/server/db/migrations/0014_add_planned_admin_menus.sql` 添加：

- 父菜单：`人才与对应科研管理`
- 子菜单：`课题时空管理`
- 路由路径：`topic-space-time`
- 前端组件：`talentResearch/topicSpaceTime/index`
- 权限标识：`talentResearch:topicSpaceTime:list`

当前页面文件：

- `chunyu-cms-admin/src/views/talentResearch/topicSpaceTime/index.vue`

页面已有的 UI 骨架：

- 搜索区：课题名称、课题级别、研究状态。
- 操作区：新增课题、批量删除。
- 表格区：多选、课题名称、课题编号、课题级别、研究状态、开始时间、结束时间、负责人、创建时间、操作。
- 弹窗表单：新增/编辑课题。
- 详情弹窗：课题基础字段展示。
- 分页组件：复用 `Pagination`。

当前不足：

- 列表数据来自 `mockData`。
- 查询条件没有真正过滤后端数据。
- 新增、编辑、删除只显示成功提示，不保存。
- 缺少后端 API、Drizzle schema、SQL migration。
- 缺少课题来源、摘要、年份、进度、状态等完整字段。
- 缺少导入、导出、启停、排序和批量操作。

### 2.2 静态页面信息

静态原型位于：

- `docs/后台静态页面-新中医大/人才与对应科研管理/课题时空管理.html`

原型表达的业务方向：

- 管理课题信息和时间节点数据。
- 顶部展示课题总数、进行中、已完成等统计。
- 支持按课题名称/编号、课题级别、状态、年份筛选。
- 列表卡片展示课题编号、级别、状态、起止时间、负责人和进度。

实施时需要注意：静态页面使用 Tailwind 圆角卡片风格，更偏展示型原型；当前真实后台是 Element Plus / RuoYi 风格。开发时不应照搬静态稿的大圆角卡片，而应把其中的业务能力转译为现有后台的搜索表单、表格、标签、进度条、弹窗和可视化面板。

### 2.3 UI 风格理解

项目后台整体是传统中后台管理系统：

- 左侧深色导航 + 顶部浅色工具栏 + 中间白色内容区。
- 页面以 `app-container`、`el-card shadow="never"`、`el-form inline`、`el-table`、`Pagination`、`el-dialog` 组织。
- 操作按钮使用 Element Plus 图标，如 `Plus`、`Search`、`Refresh`、`Delete`、`Upload`、`Download`、`View`、`Edit`。
- 状态、级别、进度等轻量信息使用 `el-tag`、`el-progress`。
- 详情使用 `el-descriptions`，长文本使用 `show-overflow-tooltip` 或详情弹窗展示。
- 交互优先级是快速检索、批量管理、低学习成本，不做营销式布局。

课题时空管理应延续专家库页面的实现方式：

- 第一屏保持“搜索卡片 + 操作按钮 + 数据表格”。
- 统计能力作为辅助面板，不抢占 CRUD 主流程。
- 表单保持两列栅格，字段多时使用 `el-tabs` 分组：基础信息、时间进度、关联信息。

## 3. 功能范围

### 3.1 MVP 必做

#### 列表与查询

- 分页查询课题列表。
- 支持按课题名称模糊搜索。
- 支持按课题编号模糊搜索。
- 支持按课题级别筛选。
- 支持按研究状态筛选。
- 支持按年份筛选，匹配 `start_year` 或 `end_year`。
- 支持按负责人筛选。
- 默认按排序升序、更新时间倒序、主键倒序展示。
- 空数据时展示 Element Plus 表格空态。

#### 新增与编辑

- 新增课题。
- 编辑课题。
- 表单字段校验。
- 编辑时回显全部基础信息和时间信息。
- 提交后刷新列表。

#### 详情

- 查看课题详情。
- 展示课题名称、编号、级别、来源、负责人、研究状态、起止时间、年份、摘要。
- 展示课题进度。
- 展示与专家关联数量，第二阶段可展示关联专家列表。

#### 删除与启停

- 单条删除。
- 批量删除。
- 删除前二次确认。
- 删除采用逻辑删除，避免专家关联数据不可追溯。
- 支持启用/停用状态切换。

#### 时间管理

- 录入开始日期、结束日期。
- 自动拆出开始年份、结束年份，用于年份筛选和统计。
- 列表展示起止时间。
- 列表展示进度条：
  - 已完成：100%。
  - 筹备中：0% 或手动进度。
  - 进行中：可优先使用手动 `progress`，没有则按当前日期和起止日期估算。
- 表单校验结束日期不能早于开始日期。

#### 导入导出预留

- 第一阶段按钮可先保留为提示。
- 后端数据结构和服务方法预留导入/导出扩展点。
- 第二阶段接入 Excel 批量导入和导出。

### 3.2 第二阶段

- 顶部统计面板：课题总数、进行中、已完成、即将到期。
- 课题导入：Excel 模板下载、批量校验、错误行提示。
- 课题导出：按当前筛选条件导出。
- 专家关联：在课题详情中展示关联专家，支持跳转专家详情。
- 课题成果关联：后续可关联论文、专利、活动或附件。
- 时间节点管理：立项、中期检查、结题、成果提交等里程碑。

### 3.3 暂不做

- 不做复杂科研项目审批工作流。
- 不在课题页面直接新增专家主数据。
- 不在课题页面直接新增论文、专利主数据。
- 不做前台展示页重构。
- 不引入新的前端 UI 框架。

## 4. 数据设计

### 4.1 `topic`

用于课题时空管理，也作为专家库关联成果之一。字段建议基于 `docs/数据库字段规划.md` 落地。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `topic_id` | `int` | 主键，自增 |
| `title` | `varchar(300)` | 课题名称，必填 |
| `topic_no` | `varchar(100)` | 课题编号 |
| `level` | `varchar(100)` | 课题级别：国家级、省部级、厅局级、校级等 |
| `source` | `varchar(200)` | 项目来源 |
| `leader` | `varchar(100)` | 负责人 |
| `research_status` | `varchar(50)` | 研究状态：筹备中、进行中、已完成 |
| `progress` | `int` | 进度，0-100 |
| `start_year` | `int` | 开始年份 |
| `end_year` | `int` | 结束年份 |
| `start_date` | `date` | 开始日期 |
| `end_date` | `date` | 结束日期 |
| `summary` | `text` | 摘要 |
| `sort` | `int` | 排序 |
| `status` | `char(1)` | 状态：`0` 启用，`1` 停用 |
| `del_flag` | `char(1)` | 删除标记：`0` 未删除，`1` 已删除 |
| 通用字段 |  | `remark/create_by/create_time/update_by/update_time` |

兼容说明：

- 当前前端 mock 使用 `level: national/provincial/municipal/school` 和 `researchStatus: preparing/ongoing/completed`。
- 数据库可继续存英文枚举，前端通过映射展示中文；也可存中文值。
- 为避免后续筛选和国际化混乱，建议落地为稳定英文枚举，前端维护 label map。

建议枚举：

| 字段 | 值 | 展示 |
| --- | --- | --- |
| `level` | `national` | 国家级 |
| `level` | `provincial` | 省部级 |
| `level` | `municipal` | 厅局级 |
| `level` | `school` | 校级 |
| `research_status` | `preparing` | 筹备中 |
| `research_status` | `ongoing` | 进行中 |
| `research_status` | `completed` | 已完成 |

### 4.2 与专家库的关系

专家库已规划 `talent_topic` 关系表：

- `talent_id`：专家 ID。
- `topic_id`：课题 ID。
- `relation_role`：关系角色，如负责人、参与人。
- `sort`：排序。

课题主表实现后，专家库的“关联课题”下拉应从真实 `topic` 表查询，不再只返回占位数据。

### 5.3 服务层规则

- 新增时校验课题名称不能为空。
- 课题编号建议唯一；如果历史数据不稳定，先做非强制唯一索引，服务层提示重复。
- 结束日期早于开始日期时返回 400。
- 自动从 `startDate/endDate` 提取 `startYear/endYear`。
- `progress` 限制在 0-100。
- 删除使用 `delFlag = '1'`。
- 列表默认过滤 `delFlag = '0'`。
- 状态只允许 `0`、`1`。
- 详情查询不存在或已删除数据时返回 404。

## 9. 风险与决策点

### 9.1 状态枚举

风险：页面已有 `researchStatus`，数据库规划里只有通用 `status`。两者语义不同。

建议：

- `status` 表示数据启用/停用。
- `research_status` 表示课题研究状态。

### 9.2 课题编号唯一性

风险：历史数据可能存在编号缺失或重复。

建议：

- MVP 表单建议课题编号必填。
- 是否加数据库唯一索引需要先确认历史数据质量。
- 若暂不加唯一索引，服务层仍可做重复提示。

## 10. 建议优先级
