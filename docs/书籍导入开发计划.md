# 书籍导入开发计划

## 1. 背景与目标

当前后台已经存在 `论文著作与成果管理 / 书籍导入` 菜单、静态设计稿和 Vue 页面骨架，但实际页面仍由 `mockData` 驱动，新增、编辑、详情、删除、批量导入和导出都没有真实接口和持久化能力。

本计划目标是把书籍导入从静态原型升级为可用的后台模块：

- 支持书籍基础信息的新增、编辑、查询、详情、删除和发布状态维护。
- 支持 Excel 批量导入书籍元数据。
- 支持导入前预览、字段校验、重复 ISBN 识别、错误行提示。
- 支持导入批次记录，便于追踪历史导入结果。
- 支持封面图片上传和回显。
- 支持书籍本体文件存入 RustFS 对象存储，数据库只保存元数据、对象 Key 和访问地址。
- 保持当前项目的 Element Plus 中后台 UI 风格和 Nuxt + Drizzle 后端实现方式。

## 2. 当前代码现状

### 2.1 菜单与页面

菜单由历史菜单迁移添加：

- 父菜单：`论文著作与成果管理`
- 子菜单：`书籍导入`
- 路由路径：`book-import`
- 前端组件：`paperAchievement/bookImport/index`
- 权限标识：`paperAchievement:bookImport:list`

当前页面文件：

- `chunyu-cms-admin/src/views/paperAchievement/bookImport/index.vue`

页面已有的 UI 骨架：

- 搜索区：书名、作者、类别、状态。
- 操作区：新增书籍、批量导入、导出、批量删除。
- 表格区：多选、书名、作者、封面、类别、出版时间、ISBN、状态、创建时间、操作。
- 弹窗表单：新增/编辑书籍。
- 详情弹窗：书籍基础字段展示。
- 批量导入弹窗：Excel 文件上传入口。
- 分页组件：复用 `Pagination`。

当前不足：

- 列表数据来自 `mockData`。
- 查询条件没有真正过滤后端数据。
- 新增、编辑、删除只显示成功提示，不保存。
- 批量导入弹窗只做静态上传控件，没有解析、预览和确认导入流程。
- 封面字段是 URL 输入框，没有复用现有上传组件。
- 导出只显示“开发中”。
- 缺少后端 API、Drizzle schema、SQL migration。
- 缺少导入批次和导入错误明细记录。

### 2.2 静态页面信息

静态原型位于：

- `docs/后台静态页面-新中医大/论文著作与成果管理/书籍导入.html`

原型表达的业务方向：

- 支持 `.xlsx`、`.xls`、`.csv` 文件上传。
- 支持下载标准模板。
- 支持导入数据预览。
- 支持识别新增、重复等导入状态。
- 展示导入历史，包括导入时间、记录数和状态。
- 展示统计数据：书籍总数、本月导入、待审核。

实施时需要注意：静态页面使用 Tailwind 大圆角卡片布局，更偏概念展示；当前真实后台是 Element Plus / RuoYi 风格。开发时不应照搬静态稿的大卡片布局，而应把其中的业务能力转译为现有后台的搜索表单、表格、弹窗、上传组件、预览表格和导入历史表格。

### 2.3 与“仿真录入”的边界

项目文档里同时存在两个概念：

- `书籍导入`：当前菜单和页面，重点是批量导入书籍元数据。
- `仿真录入` / `书籍仿真录入`：支持书籍封面、封底、核心章节图片集，用于前台仿真翻页效果。

本计划只处理 `书籍导入`：

- 做书籍基础信息和 Excel 批量导入。
- 做封面图上传和基础回显。
- 不做封底、核心章节图片集。
- 不做前台仿真翻页效果。
- 不把仿真页面的图片集管理混入书籍导入 MVP。

如果后续需要让导入后的书籍进入仿真录入流程，可以第二阶段增加“生成仿真记录”或“关联仿真记录”的入口。

### 2.4 UI 风格理解

项目后台整体是传统中后台管理系统：

- 左侧深色导航 + 顶部浅色工具栏 + 中间白色内容区。
- 页面以 `app-container`、`el-card shadow="never"`、`el-form inline`、`el-table`、`Pagination`、`el-dialog` 组织。
- 操作按钮使用 Element Plus 图标，如 `Plus`、`Search`、`Refresh`、`Delete`、`Upload`、`Download`、`View`、`Edit`。
- 状态信息使用 `el-tag`。
- 图片使用 `el-image` 或现有 `ImageUpload`。
- 详情使用 `el-descriptions`。
- 长文本使用 `show-overflow-tooltip` 或详情弹窗展示。

书籍导入应延续现有后台风格：

- 第一屏保持“搜索卡片 + 操作按钮 + 数据表格”。
- 批量导入使用弹窗流程，不做独立大屏式导入页。
- 导入预览使用 `el-table`，错误信息使用 `el-tag`、`el-alert` 和可下载错误报告。
- 表单保持两列栅格，字段多时用分组而不是营销式卡片。

## 3. 功能范围

### 3.1 MVP 必做

#### 列表与查询

- 分页查询书籍列表。
- 支持按书名模糊搜索。
- 支持按作者模糊搜索。
- 支持按 ISBN 精确或模糊搜索。
- 支持按类别筛选。
- 支持按发布状态筛选。
- 支持按出版年份筛选。
- 默认按排序升序、更新时间倒序、主键倒序展示。
- 空数据时展示 Element Plus 表格空态。

#### 新增与编辑

- 新增书籍。
- 编辑书籍。
- 表单字段校验。
- 编辑时回显基础信息和封面。
- 提交后刷新列表。

#### 详情

- 查看书籍详情。
- 展示书名、副标题、作者、ISBN、类别、出版社、出版日期、封面、简介、状态、备注。

#### 删除与状态

- 单条删除。
- 批量删除。
- 删除前二次确认。
- 删除采用逻辑删除。
- 支持发布/草稿状态切换。

#### 封面

- 复用现有 `ImageUpload` 组件上传封面。
- 存储图片 URL 或相对路径，不存二进制。
- 列表中展示小尺寸封面，详情中展示较大封面。

#### 书籍本体与 RustFS

- 书籍 PDF、EPUB 或其他数字本体文件不存数据库。
- 文件本体上传到 RustFS 对象存储。
- 数据库只保存书籍元数据、对象 Key、Bucket、文件名、文件大小、MIME 类型、访问地址和 ETag。
- 支持后台上传书籍本体文件并绑定到书籍记录。
- 支持获取书籍访问地址。MVP 可先保存可访问 URL；如果 Bucket 私有，后续改为后端生成临时签名 URL。
- 支持替换书籍本体文件，旧对象是否清理放到第二阶段。

#### 批量导入

- 下载标准 Excel 模板。
- 上传 `.xlsx`、`.xls` 文件。
- 解析 Excel 数据。
- 导入前预览解析结果。
- 校验必填字段。
- 校验 ISBN 重复。
- 校验日期格式。
- 校验类别枚举。
- 确认导入后批量写入数据库。
- 导入完成后记录导入批次。
- 导入失败时展示错误行和错误原因。

#### 导入历史

- 展示导入批次列表。
- 展示文件名、总条数、成功条数、失败条数、状态、导入时间、错误摘要。
- 支持查看某次导入的错误明细。

#### 导出

- 按当前筛选条件导出书籍列表。
- 第一阶段导出书籍基础字段，不导出图片二进制。

### 3.2 第二阶段

- CSV 导入。
- 错误报告 Excel 下载。
- 导入时支持“跳过重复 / 更新重复 / 报错终止”策略。
- 导入任务异步化，支持大文件后台处理。
- RustFS 私有 Bucket 临时访问地址。
- 书籍本体文件替换后的旧对象清理任务。
- 书籍审核状态流转。
- 与仿真录入打通：从书籍生成仿真记录，或在仿真记录中选择已有书籍。
- 前台书籍展示页或检索页联动。

### 3.3 暂不做

- 不做封底和核心章节图片集管理。
- 不做仿真翻页效果。
- 不做 OCR 自动识别书籍封面。
- 不做 ISBN 外部接口自动补全。
- 不做复杂审核工作流。
- 不做前台展示页重构。
- 不引入新的前端 UI 框架。

## 4. 数据设计

### 4.1 `book`

用于书籍基础信息管理和批量导入。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `book_id` | `int` | 主键，自增 |
| `title` | `varchar(300)` | 书名，必填 |
| `subtitle` | `varchar(300)` | 副标题 |
| `author` | `varchar(500)` | 作者，多个作者可用分隔符保存 |
| `isbn` | `varchar(100)` | ISBN，建议唯一 |
| `category` | `varchar(100)` | 类别 |
| `publisher` | `varchar(300)` | 出版社 |
| `publish_year` | `int` | 出版年份 |
| `publish_date` | `date` | 出版日期 |
| `cover_image` | `varchar(500)` | 封面图 |
| `object_bucket` | `varchar(255)` | RustFS Bucket |
| `object_key` | `varchar(500)` | RustFS 对象 Key |
| `object_url` | `varchar(1000)` | 访问地址，可为公开 URL 或后端访问地址 |
| `object_etag` | `varchar(255)` | 对象 ETag |
| `file_name` | `varchar(255)` | 原始文件名 |
| `file_size` | `bigint` | 文件大小，字节 |
| `mime_type` | `varchar(150)` | 文件 MIME 类型 |
| `storage_provider` | `varchar(50)` | 存储提供方，默认 `rustfs` |
| `summary` | `text` | 简介 |
| `source_file_url` | `varchar(500)` | 导入来源文件 |
| `sort` | `int` | 排序 |
| `status` | `char(1)` | 状态：`0` 草稿，`1` 发布 |
| `del_flag` | `char(1)` | 删除标记：`0` 未删除，`1` 已删除 |
| 通用字段 |  | `remark/create_by/create_time/update_by/update_time` |

兼容说明：

- 当前前端 mock 使用 `cover`，建议落库字段为 `cover_image`，前端可映射为 `coverImage`。
- 当前页面类别枚举为 `theory/clinical/pharmacy/acupuncture/health`，可先沿用。
- ISBN 可能存在历史缺失。建议服务层做重复校验，数据库唯一索引是否强制需要根据实际数据质量确认。
- 书籍本体文件只保存 RustFS 引用信息，不在 MySQL 中存二进制。
- `object_url` 是访问地址缓存字段，真实访问控制以 RustFS Bucket 策略或后端签名接口为准。

类别建议：

| 值 | 展示 |
| --- | --- |
| `theory` | 中医理论 |
| `clinical` | 临床实践 |
| `pharmacy` | 中药学 |
| `acupuncture` | 针灸推拿 |
| `health` | 养生保健 |

### 4.2 `import_batch`

项目字段规划已有 `import_batch`，但当前说明只覆盖论文、专利、课题。书籍导入建议复用该表，并扩展 `import_type`：

| `import_type` | 说明 |
| --- | --- |
| `1` | 论文 |
| `2` | 专利 |
| `3` | 课题 |
| `4` | 书籍 |

字段建议沿用现有规划：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `import_batch_id` | `int` | 主键 |
| `import_type` | `char(1)` | 导入类型，书籍为 `4` |
| `file_name` | `varchar(255)` | 原始文件名 |
| `file_url` | `varchar(500)` | 上传文件地址 |
| `total_count` | `int` | 总条数 |
| `success_count` | `int` | 成功条数 |
| `fail_count` | `int` | 失败条数 |
| `status` | `char(1)` | `0` 成功，`1` 处理中，`2` 失败，`3` 部分成功 |
| `error_message` | `text` | 错误摘要 |
| 通用字段 |  | `remark/create_by/create_time/update_by/update_time` |

### 4.3 `import_batch_detail`

建议新增导入明细表，用于保存错误行和预览状态。否则只能在 `error_message` 里保存摘要，不利于排错。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `import_batch_detail_id` | `int` | 主键 |
| `import_batch_id` | `int` | 批次 ID |
| `row_no` | `int` | Excel 行号 |
| `row_data` | `text` | 原始行 JSON |
| `target_id` | `int` | 成功导入后的业务 ID |
| `status` | `char(1)` | `0` 待导入，`1` 成功，`2` 失败，`3` 跳过 |
| `error_message` | `text` | 错误原因 |
| 通用字段 |  | `remark/create_by/create_time/update_by/update_time` |

MVP 可选策略：

- 如果要快速上线，可只落 `import_batch`，错误明细保存在 `error_message` JSON 中。
- 如果要可维护，建议一开始就加 `import_batch_detail`。

#### 确认导入

`POST /api/admin/paperAchievement/book/import/confirm`

请求体：

```json
{
  "batchId": 12,
  "duplicateStrategy": "skip"
}
```

`duplicateStrategy` 可选：

- `skip`：跳过重复 ISBN。
- `update`：用导入数据更新已有书籍。
- `error`：遇到重复 ISBN 则导入失败。

MVP 建议默认使用 `skip`。

- `multipart/form-data`
- 字段：`file`
- 支持类型：MVP 建议先允许 `.pdf`、`.epub`，后续按需求扩展。

处理：

- 后端上传文件到 RustFS。
- 对象 Key 统一按 `books/yyyy/MM/{uuid}.{ext}` 生成。
- 返回对象引用信息，不直接写入 `book` 表，除非请求中带 `bookId`。

返回：

```json
{
  "code": 200,
  "data": {
    "provider": "rustfs",
    "bucket": "cms-dev",
    "objectKey": "books/2026/04/uuid.pdf",
    "objectUrl": "http://127.0.0.1:9001/cms-dev/books/2026/04/uuid.pdf",
    "etag": "etag-value",
    "fileName": "伤寒论注疏.pdf",
    "fileSize": 2048000,
    "mimeType": "application/pdf"
  }
}
```

#### 绑定书籍本体

`PUT /api/admin/paperAchievement/book/object/bind`

请求体：

```json
{
  "bookId": 1,
  "provider": "rustfs",
  "bucket": "cms-dev",
  "objectKey": "books/2026/04/uuid.pdf",
  "objectUrl": "http://127.0.0.1:9001/cms-dev/books/2026/04/uuid.pdf",
  "etag": "etag-value",
  "fileName": "伤寒论注疏.pdf",
  "fileSize": 2048000,
  "mimeType": "application/pdf"
}
```

说明：

- 用于先上传对象、后绑定书籍记录的场景。
- 新增/编辑书籍也可以直接携带这些字段保存，绑定接口作为单独文件管理入口预留。

#### 获取书籍访问地址

`GET /api/admin/paperAchievement/book/:id/access-url`

返回：

```json
{
  "code": 200,
  "data": {
    "bookId": 1,
    "objectUrl": "http://127.0.0.1:9001/cms-dev/books/2026/04/uuid.pdf",
    "expiresIn": 0
  }
}
```

说明：

- Public Bucket 可直接返回 `objectUrl`。
- Private Bucket 后续改为生成短时效 presigned URL，接口路径保持不变。

### 5.3 服务层规则

- 新增时校验书名不能为空。
- 新增时校验作者不能为空。
- ISBN 非空时做重复校验。
- 出版日期合法时自动提取 `publishYear`。
- 类别只允许既定枚举；未知类别可归为 `other` 或提示错误，MVP 建议提示错误。
- 删除使用 `delFlag = '1'`。
- 列表默认过滤 `delFlag = '0'`。
- 状态只允许 `0`、`1`。
- 导入预览阶段不直接写入 `book` 表。
- 确认导入阶段才批量写入 `book` 表。
- 导入批次和导入明细需要记录原始行数据，便于后续排查。
- 书籍本体上传必须走 RustFS 对象存储服务，数据库只保存对象引用信息。
- 删除书籍默认只逻辑删除数据库记录，不立即删除 RustFS 对象，避免误删；对象清理放到第二阶段任务。

### 5.4 RustFS 对象存储预留

项目 `.env` 已存在 RustFS/S3 兼容配置：

- `PUBLIC_STORAGE_REGION`
- `PUBLIC_STORAGE_ENDPOINT`
- `PUBLIC_STORAGE_ACCESS_KEY_ID`
- `PUBLIC_STORAGE_SECRET_ACCESS_KEY`
- `PUBLIC_STORAGE_BUCKET_NAME`
- `PUBLIC_STORAGE_FORCE_PATH_STYLE`

实现建议：

- 新增 `ObjectStorageServices`，封装 RustFS S3 兼容上传、获取公开地址、生成签名地址。
- 服务层不要直接依赖具体 SDK，避免后续替换 MinIO、S3 或其他对象存储时影响业务代码。
- MVP 可以先实现 `putObject` 和 `getObjectUrl`。
- 如果 RustFS Bucket 是公开读，`objectUrl` 可按 `endpoint/bucket/objectKey` 拼接。
- 如果 RustFS Bucket 是私有读，`objectUrl` 只作为内部记录，前端访问必须走 `access-url` 接口获取临时签名地址。

### 5.5 Excel 解析策略

项目已有：

建议：

## 9. 风险与决策点

### 9.1 `book` 与 `book_simulation` 是否合并

风险：项目字段规划已有 `book_simulation`，但它语义是仿真翻页录入，不是普通书籍元数据导入。

建议：

- `book` 保存普通书籍主数据。
- `book_simulation` 保存仿真翻页配置。
- 第二阶段通过 `book_id` 关联两者，而不是一开始混表。

### 9.2 ISBN 唯一性

风险：历史数据可能存在无 ISBN 或重复 ISBN。

建议：

- ISBN 非空时服务层做重复识别。
- 是否添加数据库唯一索引，需要先确认历史数据质量。
- MVP 导入策略默认“跳过重复”。

### 9.3 导入预览数据存储

风险：导入预览如果只存在内存，刷新页面后批次丢失；如果立即写入主表，错误回滚复杂。

建议：

- 预览阶段写入 `import_batch` 和 `import_batch_detail`。
- 确认导入阶段再写入 `book`。

### 9.4 大文件导入

风险：一次性解析大 Excel 文件可能导致请求超时。

建议：

- MVP 限制文件大小和行数，例如 10MB / 5000 行。
- 第二阶段再做异步任务队列。

### 9.5 RustFS 访问策略

风险：如果直接把 RustFS 公开地址保存并暴露给前端，后续 Bucket 改为私有时会影响访问逻辑。

建议：

- 数据库保存 `object_key`、`object_bucket` 和 `object_url`。
- 前端访问优先调用 `access-url` 接口，不直接依赖 `object_url` 字段。
- MVP Public Bucket 可返回固定 URL；Private Bucket 再无缝切换为 presigned URL。

## 10. 建议优先级

1. 先做 `book` 主表、后端接口和前端真实 CRUD。
2. 再做封面上传回显和 RustFS 书籍本体上传接口。
3. 接着做 Excel 模板下载、预览和确认导入。
4. 最后做导出、导入历史和错误报告。

这样可以最快把“书籍导入”变成可录入、可查询、可批量导入的数据底座，再逐步增强导入体验和历史追踪。
